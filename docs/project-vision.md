# Project vision

Цель: создать MVP многоагентной системы, которая по текстовому описанию задачи автоматически генерирует SQL-запросы для PostgreSQL, проверяет их на уязвимости безопасности и итеративно исправляет до получения одобренного результата — с полным логом аудита для пользователя.

## Архитектура

### NL -> PostgreSQL

Реализуется через LLM, но не просто промпт, а с жескими ограничениями.

Input:
- текст запроса пользователя
- схема БД (таблица, поля, связи)
- базовые правила, например:
	- no SELECT *
	- must use WHERE for any filtering
	- only allowed tables: {allowlist}
	- max LIMIT 1000

Output JSON:
- возвращает не просто SQL а json, для более четкой передачи данных между агентами и снижения галлюцинаций:
```json
{
  "sql": "...",
  "tables_used": ["..."]
}
```

### Schema Context Agent (опционально)

В зависимости от запроса пользователя осуществляет получение из postgres схем нужных таблиц и связи между ними. Нужен, что бы не вручную каждый раз передавать схему с БД, а она автоматически прорастала в промпт модели. 

Вопрос: Нужно уточнить, есть ли смысл в этом компоненте и как правильно с точки зрения пользователя передавать в модель схему БД.

### Security Judge

Компонент, который проверяет сгенерированный SQL на соответствие правилам безопасности и корректности перед выполнением.

У нас уже есть некоторый список нарушений и уязвимостей от заказчика + есть базовые правила по работе с SQL - нужно продумать как внедрить их в агента, что бы это было удобно конфигурируемо.

Нужно реализовать статическую проверку правил, например, при помощи SQL AST parser. 

Валидация SQL запроса, что он правильно составлен синтаксически.

Дополнительно в качестве второй проверки можно использовать LLM.

Input:
- SQL-запрос
- schema context
- security policy (rules)

Output JSON:
- список нарушений и в какой строке и колонке они были найдены
- учет критичности нарушений, и их количественная оценка (из таблицы в LMS)
- summary в человекочитаемом виде

### Execution Validator (опционально)

Если мы хотим прям точно убедиться, что запрос выполняется, то можно для написанного запроса попытаться выполнить его в песочнице:
- сгенерировать по схеме таблицы
- наполнить их синтетическими данными
- выполнить SQL запрос по ним

Можно получить дополнительный список warning от команды EXPLAIN:

Примеры проверок:
- Sequential Scan на больших таблицах → warning
- отсутствие index usage → warning
- estimated rows слишком большие → fail
- nested loop join на больших данных → warning

Вопрос: спросить про такую валидацию запросов в песочнице, дееспособна ли эта идея и полезна?
### Fixer

Принимает текущий вариант запроса и список нарушений от Judge и выполняет точечные исправления. Реализован через LLM, но с четким промптом, что бы трогал только то, где были найдены ошибки. Если предоставить ему слишком много свободы, то он может повторять ошибки по кругу и галлюцинировать.

Output JSON:
- исправленный sql запрос
- список изменений в человеко читаемом виде
### Оркестратор флоу агента
Осуществляет последовательный запуск агентов. 
Реализует цикл из генерация -> валидация -> исправление. (RAG граф)
Содержит критерии остановки. (timeout, число итераций)
### Audit Logger
полный трассинг (prompts, версии SQL, замечания, решения).
### QA
- нужен пайплайн с автоматической проверкой решения (из lms требование)
- снятие метрик на test выборке (Execution Accuracy на тестовом наборе составляет не менее 70%; результат измерен и задокументирован)
- создать собственный датасет - Команда подготовила собственный датасет с примерами уязвимостей (не менее 50 пар); датасет использован для объективной оценки качества судьи
### Backend FastAPI
- для MPV синхронные запросы
- затем нужно добавить очередь, циклы проверки очень легко могут занимать 1 - 2 минуты.
- добавить streaming при получении статуса результата, что бы статус менялся в реальном времени - отзывчивый ui

Максимально тонкий: 2 ручки, очередь и streaming api. Развертывание в docker-compose.
